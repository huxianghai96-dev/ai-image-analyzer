# AI 协作说明（人机协作透明度声明）

本项目的主要代码由 AI Agent（Gemini 等）生成，但核心设计、问题定位、边界约束以及最终的模型验证与整合均由人工主导完成。

以下依据人机协作的核心维度，对协作过程进行详细说明：

## 1. 批判思维：主动发现并修正 AI 错误

在开发过程中，并非盲目接受 AI 的输出（“看起来对就用”），而是建立了**基于人工先验知识和对比测试的判断机制**：
- **AI 错误案例一（清晰度算法虚高）**：最初 AI 提供基础的 Laplacian 方差作为清晰度唯一指标。但在测试中我发现，高颗粒噪点图（如 `noise_masquerade_sharp.jpg`）被 AI 算法打出了近满分。我**主动指出并复现了该错误**，分析其根因是二阶导数无法区分噪点与真实边缘。随后我人工介入，要求引入噪点估计指标，并在 `scorer.py` 中增加了“清晰度×噪点交叉校正”的惩罚机制，才修正了这一失准问题。
- **AI 错误案例二（移动端依赖水土不服）**：AI 最初为了快速实现 HEIC 解码，直接建议将 `pillow-heif` 编入 `requirements.txt`。但我发现 Flet 在打包 Android APK 时，该库缺失对应架构的 wheel 文件会导致打包失败或崩溃。我判断这是环境依赖冲突，于是手动调整了策略，将 HEIC 作为桌面端的“动态扩展能力”，而在移动端主动捕获异常并给出友好提示。

## 2. 提问能力：结构化提问与迭代驱动

为了获得高质量的代码，我采用了**结构化、分层次的提问策略**，驱动 AI 实现了单轮无法完成的复杂系统：
- **拆解式提问**：没有一次性把几十行的需求丢给 AI，而是先问架构（“在只有 Python 且需要打包 Android 的约束下，有什么最轻量的 UI 方案？”），再问核心算法（“给出 4 种无深度学习的图像质量评估 Python 实现”），最后问整合（“如何用 Flet 将刚才的算法封装并避免阻塞主线程？”）。
- **迭代与追问**：当 AI 给出的 Flet UI 使用了路径输入框时，我持续追问：“手机上输入路径不现实，如何调用原生相册？”驱动 AI 找到了 `FilePicker` 并完成了异步状态管理的重构。

## 3. 整合能力：多模块、多模型与经验的融合

我不仅是 AI 代码的“搬运工”，更注重**系统性的拼接与自身工程经验的融合（1+1>2）**：
- **多模型能力融合**：在传统 CV 算法的基础上，我独立集成了深度学习评估模型（`pyiqa`）作为加分项扩展，并设计了 `传统分×0.5 + 深度学习分×0.5` 的混合决策树，证明了自己能将多种算法思路有机结合。同时，我还额外整合了 Gemini 的多模态 Vision API（`ai_analyzer.py`），实现了基于大模型的自然语言画质点评。
- **工程化打磨**：AI 生成的零散脚本被我通过 `src/` 和 `tests/` 目录结构重新组织。我手工编写了 `tests/run_benchmark.py` 和 `build_apk.py`，将散落的代码整合成一个包含数据验证、无痕清理、一键打包的完整工程化体系。

## 4. 关键路径中人的角色总结

- **需求解读与边界定义**：人工将作业约束拆解为明确的 4 个维度的算法选型。
- **验证体系构建**：人工设计了 3 大类反例场景（噪点伪装清晰、过度锐化光晕、雾天低对比度），填平了“能跑”到“结果可信”之间的鸿沟。
- **最终拍板**：对算法权重（35%/25%/25%/15%）的最终调节、移动端 UI 的交互逻辑排布均由人工通过实测和体验做出最后决策。
